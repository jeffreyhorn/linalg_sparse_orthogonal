# Sprint 28 Day 8 — supernodal-etree reordering: supernode-count empirical findings
#
# Day-7's interim measurement surfaced the supernode-count metric as
# the right headline contract (`non_pipeline_interim_day7.txt`): nnz_L
# is invariant by construction under symmetric permutation, so the
# real signal is whether the postorder composition produces more /
# larger fundamental supernodes per `chol_csc_detect_supernodes`.
#
# Day-8 measurement: factor each corpus fixture under env={off, on},
# convert factors.F to CholCsc via chol_csc_from_sparse, call
# chol_csc_detect_supernodes with min_size ∈ {2, 4}, report:
#   - count: number of detected fundamental supernodes meeting min_size
#   - total_grouped: sum(super_sizes) — total columns inside supernodes
#   - max: largest supernode size
#   - avg: average supernode size
#
# Captured via ad-hoc /tmp/measure_supernodes.c (not committed; used
# once and discarded).
#
# Headline finding: AMD's output is approximately etree-postordered
# on these corpus fixtures.  The postorder composition essentially
# does NOT move the supernode structure — same count ± 1, same
# total_grouped on most fixtures, and on the smaller fixtures
# (nos4, bcsstk04) it slightly REDUCES the total_grouped because
# AMD's specific perm happened to produce more compact contiguous
# supernodes than the strict postorder.
#
# This matches the Day-1 dossier's literature prior
# (pivot_decision_day1.md candidate (c) "Pres_Poisson upside per
# literature: ≤ 5% nnz reduction — primarily a numeric-factor
# optimization, not fill-reduction").  The Day-7+8 implementation
# correctly produces a postordered analysis->perm without breaking
# numeric correctness (test_supernodal_postorder_residual_unchanged
# PASSES with residual ~1e-15 on bcsstk04), but the empirical signal
# on the existing corpus does not show meaningful supernode-count
# improvement.
#
# Day-10's decision doc will record:
#   - the algorithm ships correctly behind the env-var gate
#   - the production default stays OFF (no measurable improvement on
#     the corpus + the +6-7% analyze wall cost on Pres_Poisson AMD per
#     Day-7's measurement)
#   - the literal 0.85× Pres_Poisson target is formally retired with
#     Sprint 28's empirical evidence (5 sprints + non-pipeline pivots)
#
# Day-8's CORPUS-SAFETY contract is therefore "total_grouped under
# env=on is within 25 % of env=off" (test_supernodal_postorder_no_supernode_count_regression
# in test_chol_csc.c) — not "supernode count strictly increases" which
# the empirical data does not support.

# Note on the Day-7 contract test:
# `test_supernodal_postorder_etree_contract` originally asserted
# `perm_on != perm_off` (memcmp != 0) on this fixture, which happened
# to pass.  The PR #36 review (comment 3222851170) pointed out that
# differs-from-default isn't the right contract — if AMD's output
# IS in etree-postorder, the composition is the identity and
# perm_on == perm_off remains correct behaviour.  The test was
# updated to assert the direct mathematical contract:
# perm_on[k] == perm_off[postorder_off[k]] (the exact semantic of
# apply_supernodal_postorder).  This is the load-bearing assertion;
# the supernode-count invariance reported here is downstream
# evidence that the composition doesn't break supernode contiguity
# even when it does change the column ordering.
