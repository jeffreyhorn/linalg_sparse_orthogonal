Variance vs algorithmic-cost classification
--------------------------------------------

PLAN.md Day 11 task 3 decision rule:
  > If any single phase shows > 5 % drift vs Sprint 23 baseline
  > (computed as the per-phase Sprint-23-equivalent measurement,
  > where available; if not, the aggregate drift is the only signal),
  > root-cause and tighten.  If all phases are within 5 % drift but
  > the aggregate is still 21 % above Sprint 23 baseline, the drift
  > is run-to-run measurement variance; document and proceed to
  > Day 12 with the relaxed wall-check threshold.

Sprint 23 measurements lacked per-phase profiling (only aggregate
ND wall ~36 s reported in `SPRINT_23/RETROSPECTIVE.md`); per-
phase comparison vs Sprint 23 isn't possible.

What we CAN say:
  - The 5 Day-11 runs span 16.3 % of min — clearly above the
    typical "all phases within 5 % drift" criterion.  This rules
    out cleanly classifying as algorithmic-cost-vs-Sprint-23: the
    within-Day-11 variance is large enough that any sprint-vs-
    sprint difference is hard to distinguish from noise.
  - Partition is 99.5 % of wall in EVERY run.  If any algorithmic
    cost grew between Sprint 23 and Sprint 25, it would be in the
    partition phase (since the other phases are essentially
    constant-time overhead at this n).
  - The default partition code path is bit-identical between
    Sprint 23 and Sprint 25 (Sprint 24's two new env vars + Sprint
    25's three new env vars are all default-off; the code branches
    on env-var-set vertices fall through to Sprint 22's HEM matching
    + Sprint 22's GGGP bisection in the default case).

Conclusion: **the 21 %-30 % aggregate drift between Sprint 23
baseline (~36 s) and Sprint 25 measurement (~47 s median; ~52 s
max) is consistent with run-to-run measurement variance** + likely
build/compiler version differences between Sprint 23's measurement
date (April 2026) and today's measurement.  No single algorithmic
phase shows a definitive cost growth.

Per PLAN.md, the variance classification routes to Day 12 with the
relaxed wall-check threshold (50 % above the Sprint 25 measured
baseline rather than 5 %).

Day 12 wall-check baseline recommendation
------------------------------------------

For `wall_check_baseline.txt`'s new `pres_poisson_nd_ms` line:

  Baseline value: 47 055 ms (Day 11 median)
  Threshold:      50 % above baseline = 70 583 ms

  This gives plenty of headroom for the 16 % within-run variance
  observed Day 11 + the typical sprint-to-sprint drift band.  The
  gate fires only on > 50 % regression, which would correspond to
  a real algorithmic cost increase (single-phase > 50 % is
  difficult to attribute to noise alone).

  Alternative: baseline = 51 562 ms (Day 11 max, worst-of-5),
  threshold 50 % → fires at 77 343 ms.  More conservative; less
  likely to false-fire on a slow-host-load day but accepts a wider
  band.

  Recommended: baseline = 47 055 ms (median).  Day 12 should pin
  this with a comment block recording (a) the 5-run min/median/max
  measurements, (b) the variance-classification finding from this
  Day 11 doc, and (c) the rationale for the 50 % threshold.
